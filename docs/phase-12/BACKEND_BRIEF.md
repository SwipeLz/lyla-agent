# Phase 12 — Backend Brief: Observability Dashboard + Simple Auth

## Goal

Make troubleshooting **trivial** before ESP physical integration. Saat alat gagal di demo, dashboard harus bisa langsung tunjukkan: gagal di layer mana, parameter apa yang nyangkut, kapan terjadi.

**Threat model: internet-facing.** Backend dihost di server publik (VPS atau Cloudflare Tunnel ke WSL), bukan LAN demo. ESP32 nembak ke domain publik. TLS dihandle di luar app (tunnel / reverse proxy) — app tetap HTTP. Konsekuensi: auth harus aman secukupnya untuk public exposure (hashed password, default device-token gate ON, login rate-limit, cookie `Secure`), tapi tetap stdlib-only dan tanpa over-engineering.

Priority order:
1. **Observability layer** — request tracing, stage timings, error layer detection (55% effort)
2. **Internet-safe auth** — 1 user dummy login, scrypt-hashed, rate-limited (20% effort)
3. **Device pairing flow** — dashboard generates `/sd/config.json` snippet untuk operator copy ke SD (25% effort)

This brief is intentionally minimal. Prometheus akan turn ini menjadi execution plan dengan task breakdown detail.

## Scope (In)

- Single Alembic migration: extend `VoiceCommandLog` with telemetry fields + `Device` with telemetry cols
- 4 new observability endpoints (trace, recent, stats, devices)
- 3 new auth endpoints (login, logout, me) — env-driven creds, **scrypt-hashed** password, no DB table
- In-memory login rate limiter (5 fails / 5 min / IP)
- 1 new pairing endpoint (POST /devices/pair) returning ready-to-paste config blob
- `X-Device-Token` enforcement gate (`REQUIRE_DEVICE_TOKEN` setting, **default `true`**)
- Cookie-based session with `HttpOnly`, `SameSite=Lax`, `Secure` (toggle via `COOKIE_SECURE`)
- New frontend route `/observability` (live tail + drill-down + device grid)
- New frontend route `/login`
- Extend `/devices` page with "Pair New Device" modal
- Extend `_agent_helpers.py` to capture stage timings into VoiceCommandLog metadata
- CLI helper `python -m scripts.hash_dashboard_password` to generate the env value

## Scope (Out, defer)

- Multi-user support
- Plaintext password compare (replaced by scrypt; keep old behavior out of code)
- bcrypt/argon2/passlib (stdlib `hashlib.scrypt` is sufficient)
- JWT / OAuth / SSO
- Email notifications, Slack alerts
- Prometheus / Grafana metrics export
- BLE / captive-portal provisioning
- HTTPS code in app (TLS termination is external — tunnel / reverse proxy handles cert)
- `POST /devices/{id}/rotate-token` endpoint (manual procedure for 1-device MVP)
- CSRF tokens (mitigated by `SameSite=Lax` cookie + scope of dashboard endpoints)
- Long-term log retention strategy

## Auth (env-driven, single user, scrypt-hashed)

```
Settings (.env):
  DASHBOARD_USERNAME=admin
  DASHBOARD_PASSWORD_SCRYPT=<salt_hex>:<hash_hex>   # generated by helper script
  SESSION_TTL_HOURS=24
  COOKIE_SECURE=true
  REQUIRE_DEVICE_TOKEN=true
  LOGIN_RATE_LIMIT_MAX_FAILS=5
  LOGIN_RATE_LIMIT_WINDOW_SECONDS=300
```

Generate the password hash once with the bundled helper, then paste the result into `.env`:

```bash
python -m scripts.hash_dashboard_password --password admin
# prints: <salt_hex>:<hash_hex>
```

Hashing uses stdlib `hashlib.scrypt` with `n=2**14, r=8, p=1, dklen=64, salt=32 bytes`. No new dependency. ~50ms per verify on modern CPU; login is rare so latency is irrelevant.

If `DASHBOARD_PASSWORD_SCRYPT` is empty, login is **fail-closed** — every attempt returns 401 regardless of input. This is the safe default when `.env` is misconfigured.

Endpoints:

| Method | Path | Behavior |
|---|---|---|
| POST | /auth/login | body `{username, password}` → rate-limit check → username `compare_digest` → scrypt verify → if match: 200 + Set-Cookie `lyla_session=<token>` HttpOnly + Secure + SameSite=Lax; else 401 |
| POST | /auth/logout | clear cookie |
| GET | /auth/me | 200 if cookie valid, 401 if not |

Login rate-limit: in-memory `dict[ip, list[failure_timestamps]]` + `threading.Lock`. After 5 failures within 300s from the same IP, return 429. Successful login resets the counter. IP resolution prefers first hop of `X-Forwarded-For` (Cloudflare Tunnel / reverse proxy adds it) and falls back to `request.client.host`.

Session storage: in-process dict `{token: Session}`, evicted lazily. No Redis, no DB. Server restart = re-login.

Auth dependencies (FastAPI `Depends`) protect `/dashboard/*` and `/observability/*`. `/agent/*` and `/devices/{code}/*` are gated by `X-Device-Token` enforcement (default ON).

Why scrypt instead of plaintext: backend is internet-facing. Even though `.env` is gitignored, an attacker with read access to log dumps, backups, or memory snapshots should not get a usable plaintext password. scrypt cost makes brute-force impractical without giving up the "no new dependency" rule.

Why no rotate-token endpoint: 1-device MVP. Rotating requires re-flashing the SD card anyway, so a dashboard endpoint adds API surface without saving operator effort. Manual procedure: regenerate via DB UPDATE or pair-ulang via dashboard, then update SD card by hand.

## Device pairing

Single endpoint:

```
POST /devices/pair
  body: {"name": "Lyla Demo Unit"}
  auth: session cookie required
  response 201:
    {
      "device_id": "...",
      "device_code": "TASKBOT-XXXX",
      "api_token": "tk_live_...",
      "config_json": {
        "user_id": "...",
        "device_id": "...",
        "device_code": "TASKBOT-XXXX",
        "device_token": "tk_live_...",
        "base_url": "https://lyla.example.com",
        "wifi": {"ssid": "", "password": ""},
        "firmware_version": "0.1.0"
      }
    }
```

`user_id` resolves to the demo user automatically (single-user MVP). `base_url` comes from `BASE_URL` setting — set this to the public URL of your tunnel/VPS, e.g. `https://lyla-demo.trycloudflare.com`.

Frontend `/devices` page extension:
- "Pair New Device" button → modal with name input → submit → server response shown in `<textarea readonly>` + copy-to-clipboard button + instructions text "Save as `/sd/config.json` on your ESP SD card, fill in the WiFi credentials, then power on"

**No `POST /devices/{device_id}/rotate-token` endpoint.** Token rotation is intentionally manual for the 1-device MVP. If a token is compromised, the operator runs the pair flow again and replaces the SD card. Adding a rotate endpoint forces the same SD-card replacement, so the API surface is pure overhead.

## Frontend additions

### `/login` route

Form: username/password → `POST /auth/login`. On 401: error message. On 200: redirect to `/observability`.

### `/observability` route (default after login)

Layout: 3 panels.

**Top bar (4 stat cards):**
- Requests/min (last 5 min)
- p95 latency (last 1h)
- Success rate (last 1h)
- Active devices (heartbeat in last 60s)

**Middle: live tail table** (auto-refresh every 3s)

| Time | Device | audio_code | Status | Total ms | STT ms | Agent ms | TTS ms | Click |
|---|---|---|---|---|---|---|---|---|

Click row → drill-down sidebar:
- Stage timing horizontal bar chart (validate/stt/agent/classify/tts)
- Full transcript (input_text)
- Full reply (response_text)
- Parsed actions JSON (collapsed)
- Client telemetry (firmware, RSSI, battery, recording_duration)
- Error block (if any) with `layer` highlighted

**Bottom: device grid** (separate tab or sub-section)

Card per device:
- Device code + name
- Online/offline pill (green if last_seen < 60s)
- Firmware version
- WiFi RSSI bar
- Battery icon
- Last seen relative time
- Last command result (audio_code + status)

Add "Pair New Device" button next to existing list. Click → modal.

## Acceptance criteria

1. Operator can login with username + scrypt-hashed password sourced from `.env`
2. 5 failed login attempts from same IP within 5 min: 6th attempt returns 429
3. Live tail shows new requests within 3s of arrival
4. Drill-down shows clear stage breakdown — operator can say "fail at STT" or "fail at agent" within 5 seconds of looking
5. Device grid shows red/green status; offline detected within 60s
6. Pair flow: 1 click → copy config → save to SD → ESP works
7. With `REQUIRE_DEVICE_TOKEN=true` (default): `/agent/audio` without `X-Device-Token` returns 401
8. With `DASHBOARD_PASSWORD_SCRYPT=""`: every login attempt returns 401 (fail-closed)
9. Cookie set with `Secure=true` when `COOKIE_SECURE=true`
10. All 256 existing tests still pass + ~33 new tests for endpoints (≥289 total)
11. Migration is reversible (Alembic downgrade test)
12. No new entry in `requirements.txt`

## Architecture compliance (must enforce)

- Layer rules per `app/AGENTS.md`: api → agent → tools → services → models. Observability endpoints stay in `app/api/observability.py`, no business logic.
- AR7 (audio hermeticity): no provider SDK in `app/audio/*`. Phase 12 doesn't touch this.
- VoiceCommandLog append-only: `metadata_json` written ONCE during request handling, never updated.
- `/agent/text` and `/agent/audio` behavior MUST stay bit-for-bit unchanged.
- New tests follow `app/tests/AGENTS.md` conventions (network kill-switch, bounded Hypothesis ranges).

## Anti-patterns to avoid

- Pulling Prometheus/Grafana stack — overkill for 1-device demo
- WebSocket for live tail — polling 3s is fine, simpler
- Storing session in DB — in-memory dict OK for single-process MVP
- bcrypt/argon2/passlib — stdlib `hashlib.scrypt` is sufficient and zero-dep
- Plaintext password compare — internet-facing → unsafe even with gitignored `.env`
- HTTPS code in app — TLS is external (tunnel / reverse proxy) so the app can stay focused
- Multi-user RBAC — single-admin scope, defer
- Audit log of admin actions — defer
- Adding `POST /devices/{id}/rotate-token` endpoint — manual procedure simpler for 1-device MVP
- CSRF token plumbing — `SameSite=Lax` cookie + scoped endpoints sufficient for MVP
- Encrypting `device_token` at rest — gitignored `.env` is enough; tunnel terminates TLS
